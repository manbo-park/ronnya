# 리치마작 트레이닝 앱 — 점수 계산 연습 기능 구현 계획

## 1. 개요

화료(和了) 시점의 손패·후로·오름패와 상황 정보를 보고 사용자가 점수를 맞추는 연습 게임을 구현한다. 문제는 매번 랜덤 생성되며, 룰과 점수 계산은 **작혼(雀魂) 기준**을 따른다. 본 문서는 첫 번째 기능인 점수 계산 연습에 한정한 계획이며, 이후 나니키루 등 추가 기능을 확장할 수 있는 구조를 전제로 한다.

### 범위 (v1)

| 포함 | 제외 (추후 검토) |
|---|---|
| 4인 동남전 기준 상황 생성 | 3인 마작 |
| 일반형 / 치또이츠 / 국사무쌍 화료형 | 연장(本場) — 항상 0본장 |
| 리치 / 더블리치, 일발 | 하이테이·호우테이·창깡·영상개화 등 상황역(엔진만 구현, 생성 제외) |
| | 뒷도라 (작혼 룰과의 의도적 차이 — 학습 편의상 미적용, 앱 내 룰 안내에 명시) |
| 적도라 3장 (각 수패 5 한 장씩) | 공탁 리치봉 정산 |
| 깡(안깡/명깡) 및 깡도라 | 머리하네·더블론 등 복수 화료 처리 |
| 역만 (복합·더블역만·헤아림역만 포함) | 유국 관련 처리 |

## 2. 기술 스택 및 아키텍처

### 스택 제안

- **형태**: 모바일 우선 반응형 웹앱(PWA). 설치 없이 접근 가능하고, 추후 Capacitor 등으로 네이티브 패키징이 가능해 초기 비용이 가장 낮다.
- **프론트엔드**: React + TypeScript. 점수 계산처럼 케이스가 많은 도메인은 타입 안정성이 버그를 크게 줄여준다.
- **상태 관리**: 문제 1건의 상태 머신(도전 → 해답)이 전부이므로 경량 도구(Zustand 또는 useReducer)로 충분.
- **백엔드**: 불필요. 문제 생성과 채점 모두 클라이언트에서 수행(오프라인 동작 가능). 추후 통계/랭킹이 필요해지면 도입.

### 레이어 구조

```
src/
├── core/                  # 순수 로직 (UI 의존성 없음, 추후 나니키루에서 재사용)
│   ├── tiles/             # 패 정의, 정렬, 표기
│   ├── hand/              # 화료형 판정 (멘쯔 분해, 치또이, 국사)
│   ├── scoring/           # 역 판정, 부 계산, 점수 테이블
│   └── generator/         # 문제 생성기
├── features/score-trainer/ # 점수 연습 화면 (도전/해답 Phase)
└── components/             # 패 렌더링 등 공용 UI
```

`core`는 React에 의존하지 않는 순수 TypeScript로 작성한다. 패 표현·화료형 분해 로직은 나니키루 기능에서도 그대로 쓰이므로 처음부터 분리해 둔다.

## 3. 도메인 모델

### 3.1 패 (Tile)

```ts
type Suit = 'm' | 'p' | 's' | 'z';          // 만/통/삭/자
interface Tile {
  suit: Suit;
  rank: number;        // 수패 1~9, 자패 1~7 (동남서북백발중)
  isRed: boolean;      // 적도라 (5만/5통/5삭 각 1장)
}
```

- 전체 패산은 136장. 생성기와 검증기는 항상 **패 종류별 사용 매수 ≤ 4장**(적5는 수트당 1장)을 보장해야 한다.
- 정렬 규칙: 만 → 통 → 삭 → 자(동남서북백발중), 수패는 오름차순. 적5는 일반 5와 동일 위치에 정렬.

### 3.2 후로 (Meld)

```ts
type MeldType = 'chi' | 'pon' | 'minkan' | 'ankan';
interface Meld { type: MeldType; tiles: Tile[]; }
```

v1에서는 어느 플레이어로부터 울었는지(출처 방향)는 점수에 영향이 없으므로 모델링하지 않는다. 단, **안깡은 후로 영역에 표시되지만 멘젠을 깨지 않는다**는 점을 역 판정에서 구분해야 한다.

### 3.3 문제 (Problem)

```ts
interface Problem {
  hand: Tile[];            // 순수 손패 (오름패 제외)
  melds: Meld[];
  winningTile: Tile;
  winType: 'tsumo' | 'ron';
  roundWind: 1 | 2;        // 장풍: 동/남
  seatWind: 1 | 2 | 3 | 4; // 자풍: 동남서북 (동=친)
  riichi: 'none' | 'riichi' | 'double';
  ippatsu: boolean;
  doraIndicators: Tile[];      // 1 + 깡 수 (뒷도라는 미적용)
}
```

`손패 매수 + 후로×3 + 오름패 = 14`(깡은 4장이지만 멘쯔 1개로 계산)가 항상 성립해야 한다.

### 3.4 채점 결과 (ScoringResult)

```ts
interface ScoringResult {
  yaku: { name: string; han: number }[];   // 도라/적도라 포함
  totalHan: number;
  fu: number;
  fuDetails: { reason: string; fu: number }[];  // 부수 상세 (해답 화면용)
  limitName?: string;       // 만관/하네만/배만/삼배만/역만/n배역만
  payment: Payment;          // 아래 참조
}
type Payment =
  | { kind: 'ron'; total: number }
  | { kind: 'tsumoDealer'; each: number }            // 친 쯔모: x올
  | { kind: 'tsumoNonDealer'; dealer: number; others: number }; // 자 쯔모
```

## 4. 점수 계산 엔진

### 4.1 화료형 분해

- 14장(깡 포함 환산)을 **4멘쯔 + 1또이쯔**로 분해. 복수 분해가 가능한 경우(예: 핑후/탕야오 해석 차이, 슌쯔/커쯔 중복 해석) **모든 분해를 열거한 뒤 최고 점수 해석을 채택**한다(고점법).
- 치또이츠(7대자), 국사무쌍은 별도 패턴으로 판정. 치또이와 량페코가 동시에 성립하는 형태는 량페코(멘쯔수 분해) 해석이 우선되는지 점수 비교로 자동 결정된다.
- 대기 형태(양면/간짱/변짱/단기/샤보)도 분해별로 산출 — 부수 계산과 핑후 판정에 필요.

### 4.2 역 판정

지원 역 목록 (작혼 기준):

- **1판**: 리치, 일발, 멘젠쯔모, 핑후, 탕야오(울어도 인정 — 쿠이탕 있음), 이페코, 역패(백/발/중/장풍/자풍), 하이테이·호우테이·창깡·영상개화(v1 생성 제외, 엔진은 구현)
- **2판**: 더블리치, 치또이츠(25부), 또이또이, 산안커, 산깡쯔, 삼색동각, 혼노두, 소삼원, 삼색동순(울면 1판), 일기통관(울면 1판), 찬타(울면 1판)
- **3판**: 량페코, 준찬타(울면 2판), 혼일색(울면 2판)
- **6판**: 청일색(울면 5판)
- **역만**: 국사무쌍, 스안커, 대삼원, 자일색, 녹일색, 청노두, 스깡쯔, 구련보등, 소사희, 대사희, 천화·지화(v1 생성 제외)
- **더블역만** (작혼 채택): 국사 13면 대기, 스안커 단기, 대사희, 순정구련보등
- **도라류**: 도라, 적도라 (뒷도라 미적용) — 역이 아니므로 단독 화료 불가 조건에 포함하지 않음

### 4.3 부수 계산

| 항목 | 부수 |
|---|---|
| 기본부 | 20 |
| 화료 방식 | 멘젠 론 +10, 쯔모 +2 (핑후 쯔모 제외) |
| 커쯔/깡 | 중장 밍커 2 / 노두·자패 밍커 4 / 중장 안커 4 / 노두·자패 안커 8 / 명깡 8·16, 안깡 16·32 |
| 또이쯔 | 역패 +2, **연풍패(장풍=자풍) +4 (작혼 기준)** |
| 대기 | 간짱·변짱·단기 +2 |
| 특례 | 핑후 쯔모 = 20부 고정 / 치또이 = 25부 고정 / 울고 20부가 되는 형태는 30부로 처리(쿠이핑후형) |
| 올림 | 10부 단위 절상 |

부수 계산 과정은 항목별 배열(`fuDetails`)로 보존하여 해답 Phase에 그대로 노출한다.

### 4.4 점수 산출

- 기본점 = 부 × 2^(2+판), 만관 이상은 테이블 처리.
- **키리아게 만관 적용 (작혼 기준)**: 4판30부, 3판60부는 만관으로 절상.
- 5판 만관 / 6~7판 하네만 / 8~10판 배만 / 11~12판 삼배만 / 13판+ 헤아림역만. 역만 복합은 합산(예: 스안커 단기 + 자일색 = 3배역만).
- 지불 형태: 론은 단일 지불액, 친 쯔모는 "x올(all)", 자 쯔모는 "자 x / 친 y". 100점 단위 절상.

## 5. 문제 생성기

### 5.1 기본 전략: 구성적 생성 + 엔진 검증

무작위 14장을 던져 화료형이 나올 때까지 돌리는 방식은 비효율적이고 분포 제어가 안 되므로, **화료형을 역방향으로 조립**한다.

1. **패턴 선택**: 일반형 / 치또이츠 / 국사무쌍을 가중치로 선택 (예: 92% / 6% / 2%).
2. **멘쯔 조립**: 일반형이면 슌쯔/커쯔를 가중치로 4개 + 또이쯔 1개 생성. 이때 전역 패 카운터로 4장 초과를 즉시 차단.
3. **후로 결정**: 0~3개 멘쯔를 후로로 전환(치/퐁), 커쯔 일부를 깡으로 승격(안깡/명깡). 후로 수는 멘젠 비중이 높도록 가중(예: 멘젠 55%, 1후로 25%, 2후로 13%, 3후로 7%).
4. **오름패 결정**: 손패(후로 제외) 중 하나를 오름패로 지정. 대기 형태가 자연스럽도록 멘쯔 구성 요소 또는 또이쯔에서 선택.
5. **상황 부여**: 장풍/자풍, 쯔모/론, 리치 여부(멘젠일 때만, 더블리치·일발은 저확률), 적도라 치환(5 보유 시 확률 적용).
6. **도라 결정**: 도라표시패 1 + 깡 수만큼 추가. 표시패도 전역 4장 제한에 포함.
7. **엔진 검증**: 완성된 문제를 **실제 채점 엔진에 통과**시켜 (a) 화료형 성립, (b) **역 1개 이상**(도라 제외), (c) 패 매수 제한을 확인. 실패 시 재생성(리트라이 루프). 생성기와 채점기가 같은 엔진을 쓰므로 둘 사이의 불일치가 원천 차단된다.

### 5.2 합법성 제약 체크리스트

생성된 문제가 위반해서는 안 되는 조건들. 검증 단계에서 전수 확인한다.

- 모든 패 종류 사용 매수 ≤ 4 (손패 + 후로 + 오름패 + 도라표시패 합산), 적5는 수트당 ≤ 1
- 역 없는 화료(도라만) 생성 금지
- 후로(치/퐁/명깡)가 있으면: 리치·더블리치·일발·멘젠쯔모·핑후·이페코·량페코 불가. **안깡만 있는 경우 리치 허용(확정)** — 생성 시점에 안깡을 먼저 확정한 뒤 리치를 부여하므로 "리치 후 안깡의 대기 변화" 같은 모순 케이스는 발생하지 않음
- 일발은 리치 상태에서만, 더블리치와 일발 동시 성립 가능
- 치또이/국사는 후로 불가, 국사는 도라 외 형태 제약 준수
- 핑후 성립 시 부수 모순 금지 (멘젠 론 30부 / 쯔모 20부 강제)
- 자풍이 동(친)인지에 따라 지불 테이블 분기 정확성
- 도라표시패 수 = 1 + 깡 수
- (선택적 현실성 검증) 론 화료 시 명백한 후리텐 — 대기패 중 하나가 이미 손패에 4장 노출되는 등의 불가능 대기 — 방지

### 5.3 출제 분포 튜닝

순수 랜덤 조립은 또이또이·자패 비중이 실전보다 높게 나오기 쉽다. 슌쯔:커쯔 가중치, 자패 출현율, 후로율을 파라미터화해 두고, 초기값은 "30~40부 2~4판대가 과반, 만관 이상 ~15%, 역만 ~1%" 수준을 목표로 조정한다. 추후 난이도 선택(예: 부수 계산이 까다로운 문제만) 기능의 기반이 된다.

## 6. UI / UX

### 6.1 공통

- 모바일 세로 화면 기준 설계. 패 14장 + 후로가 한 줄에 들어가도록 패 크기를 화면 폭 기반으로 산출 (대략 `min(7vw, 40px)` 수준에서 조정).
- 패 렌더링: SVG/이미지 스프라이트 기반 마작패 컴포넌트. 적5는 빨간 숫자로 구분.
- Phase 상태 머신: `challenge → answer → (리셋) → challenge`.

### 6.2 도전 Phase

```
┌──────────────────────────────┐
│ 장풍: 동  자풍: 남   [쯔모]   │ ← 상황 배지 (리치/일발도 배지로)
│ 도라표시: 🀇                  │
├──────────────────────────────┤
│ 손패(정렬)   후로   오름패     │ ← 그룹 간 여백으로 구분,
│ 🀇🀈🀉 … ▏🀙🀙🀙 ▏🀟        │    오름패는 색/테두리로 강조
├──────────────────────────────┤
│ 점수 입력: [____] 점  [확인]  │
│                      [리셋]  │
└──────────────────────────────┘
```

- 정렬: 만→통→삭→자(동남서북백발중), 수패 오름차순. 후로는 손패 우측에 멘쯔 단위로 묶어 표시, 오름패는 맨 우측에 별도 여백 + 시각적 강조.
- **점수 입력 형식**: 
  - 론 → 지불 총액 1개 필드.
  - 친 쯔모 → "x올" 1개 필드.
  - 자 쯔모 → "자 지불 / 친 지불" 2개 필드 (작혼 표기와 동일하게). 
  - 숫자 키패드(`inputmode="numeric"`) 사용, 100점 단위 빠른 입력을 위한 보조 버튼(+1000/+100 등)은 v1.1 검토.
- **입력 모드 옵션 (설정에서 전환)**:
  - 기본 모드: 점수만 입력.
  - 판·부 모드: 점수 입력 필드 위에 판/부 입력 필드 추가 (부는 10부 단위 선택형 — 20/25/30/40/50/60/70…, 만관 이상은 부 입력 생략 가능 처리). 채점 시 판·부·점수 각각의 정오를 개별 표시.
- 확인 버튼 → 해답 Phase 전환. 리셋 버튼 → 즉시 새 문제.

### 6.3 해답 Phase

- 상단에 **정답/오답 판정**과 정답: `x판 y부 — z점` (쯔모는 지불 분배 형태로, 예: "4판 30부 — 만관(키리아게) 2000/3900"… 키리아게 적용 시 그 사실 표기). 판·부 입력 모드에서는 판/부/점수 각각의 정오를 개별 표시.
- **역 상세**: 역 이름 + 판수 목록 (리치 1 / 멘젠쯔모 1 / 도라 2 …).
- **부수 상세**: `fuDetails`를 그대로 테이블로 — 기본 20 + 중장 안커 4 + 간짱 대기 2 + 쯔모 2 = 28 → 절상 30부. 핑후 쯔모 20부/치또이 25부 고정 케이스는 그 사유를 명시.
- 도전 Phase의 패 표시는 그대로 유지(스크롤 없이 대조 가능하게), 하단에 리셋(다음 문제) 버튼.

## 7. 테스트 전략

점수 엔진의 정확성이 이 앱의 존재 가치이므로 테스트에 비중을 둔다.

1. **골든 테스트**: 작혼 실전 스크린샷/기보 기반으로 채집한 케이스 100+건(부수 경계 케이스 위주: 핑후쯔모 20부, 치또이 25부, 쿠이핑후형 30부, 연풍또이쯔 4부, 키리아게 만관, 역만 복합 등)을 기대값 고정 테스트로 작성.
2. **교차 검증**: 오픈소스 채점기(예: riichi 계열 라이브러리)와 랜덤 생성 문제 수만 건을 자동 대조. 불일치 케이스는 작혼 룰 차이인지 버그인지 분류.
3. **속성 기반 테스트**: 생성기가 만든 문제 N건에 대해 5.2의 합법성 제약을 전수 자동 검증.
4. **UI 테스트**: Phase 전환, 입력 형식(론/친쯔모/자쯔모 분기), 정렬 규칙 스냅숏 테스트.

- `core/` 의 패·분해 로직은 나니키루 기능에서 재사용 (수패 효율 계산기는 별도 모듈로 추가).

- 사용자 통계(정답률, 부수 계산 취약 유형) — 백엔드 도입 시점.

## 9. 확정 사항

1. **뒷도라 미적용** — 리치 문제에서도 뒷도라는 생성·계산하지 않는다. 작혼 룰과의 의도적 차이이므로 앱 내 룰 안내에 명시한다.
2. **판·부 입력 모드 옵션 제공** — 기본은 점수만 입력, 설정에서 판·부 입력 모드로 전환 가능 (6.2 참조).
3. **상황역은 생성 제외** — 하이테이·호우테이·창깡·영상개화·천화·지화는 채점 엔진에는 구현하되 문제로는 출제하지 않는다.
4. **리치 + 안깡 조합 허용** (5.2 참조).
